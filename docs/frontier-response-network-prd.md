
# Frontier Response Network PRD

- Status: Draft v0.1
- Date: 2026-03-16
- Audience: Builder / Product / Frontend / Smart Contract
- Language: Chinese

## 1. 项目一句话

`Frontier Response Network` 是一个面向 `EVE Frontier` 的外部服务网络，用来处理三类真实玩家需求：

1. `SOS Rescue`：玩家卡在野外，缺油、过热、回不去，需要别人救援。
2. `Escort`：玩家要运高价值货物穿越危险区域，需要有人护航。
3. `Recovery`：玩家死亡或战斗后，需要队友或雇佣兵回收残留资产并协助重建。

它不是“又一个地图站”或“又一个库存面板”，而是一个把 `求助 -> 接单 -> 执行 -> 结算 -> 留下声誉` 做成闭环的服务市场。

## 2. 先说作用

这份文档优先回答一个最实际的问题：

`这个东西到底有什么用？`

答案是：

它让 EVE Frontier 里原本只能靠 Discord 喊人、聊天私聊、手动转账解决的“救人、护航、回收”行为，变成一个可以公开发布、被别人接单、完成后自动结算的系统。

用最白话的方式讲：

- 你在野外飞不回来了，不用去群里刷屏求人。
- 你要运贵重货，不用先相信陌生人再碰碰运气。
- 你刚爆船，不用靠口头承诺找人帮你回收装备。

你直接发一个工单，别人接单，系统跟踪状态，完成后给钱。

## 3. 为什么这不是伪需求

这个项目不是空想，它建立在 EVE Frontier 已经明确存在的机制之上：

1. `Fuel` 用完会让玩家被困在当前区域。
2. `Heat` 过高会让系统过热，影响航行和跳跃。
3. `Death` 会带来真实损失，船会丢，货也可能丢。
4. 官方文档明确提到盟友支援和资产回收的重要性。
5. `Fleet`、`Map`、`Assets`、`Wallet` 等信息已经存在，但还没有一个统一的“服务调度台”。

也就是说，痛点本来就在游戏里，只是当前解决方式更像“社交喊人 + 表格 + 信任”，而不是产品。

## 4. 项目目标

### 4.1 业务目标

1. 让玩家能在 `3` 分钟内发布一个清晰的救援或护航请求。
2. 让其他玩家能快速筛选、接单并开始执行。
3. 让报酬结算从“私下转账”变成“公开工单 + 系统状态 + 自动放款”。
4. 把服务历史变成可见记录，逐步形成声誉。

### 4.2 比赛目标

1. 做一个评委一看就懂的真实场景，而不是抽象 infra。
2. 呈现真实玩家之间的协作与交易行为。
3. 支持 `Stillness` 上的 live demo。
4. 同时覆盖 `Utility`、`Creative`、`Live Frontier integration` 的叙事。

## 5. 非目标

以下内容不纳入 MVP：

1. 不做完整公会管理系统。
2. 不做复杂 AI 自动派单。
3. 不做链上完全自动验证的全流程裁决系统。
4. 不做语音、聊天、社交 feed。
5. 不做另一个大而全地图站。

## 6. 产品定位

### 6.1 产品本质

这是一个 `服务市场 + 结算系统`，不是一个单纯数据查询工具。

### 6.2 和现有工具的区别

现有社区工具更多是：

- 地图
- 路线规划
- 日志分析
- 仓储查看

本项目关注的是：

- 玩家求助
- 别人接单
- 双方履约
- 报酬结算

### 6.3 MVP 聚焦策略

虽然产品最终包含 `Rescue`、`Escort`、`Recovery` 三类工单，但 MVP 只打透一个故事：

`一个玩家被困 -> 发布 SOS -> 另一名玩家接单并救援 -> 请求人确认 -> 奖励自动结算`

只要这个流程打通，产品就成立。

## 7. 目标用户

### 7.1 求助者

1. 被困玩家：没油、过热、飞不回去。
2. 运输玩家：有贵重 cargo，需要护航。
3. 战斗后幸存者：资产散落，需要回收。

### 7.2 服务提供者

1. 附近愿意接单的独狼玩家。
2. 专做救援/护航的职业玩家。
3. 提供响应服务的 Tribe 或小队。

### 7.3 次要用户

1. 评委
2. 社区投票参与者
3. 招聘方

这些人虽然不是终端用户，但会通过 demo 判断这是不是一个真实产品。

## 8. 核心使用场景

### 8.1 场景 A：SOS Rescue

玩家 A 在野外卡住，跳跃燃料不足，周边区域危险。

玩家 A 进入外部工具后：

1. 连接钱包。
2. 选择当前所在系统或手动填写位置。
3. 选择求助类型：`Fuel Rescue`。
4. 填写需求：需要多少燃料、是否需要护送返航、截止时间、赏金。
5. 发起请求，并把赏金锁进 escrow。

玩家 B 看到请求后：

1. 查看距离、赏金、风险等级、截止时间。
2. 点击接单。
3. 前往目标区域。
4. 完成补给或护送。
5. 请求人确认完成。
6. 系统自动放款给 B。

### 8.2 场景 B：Escort

玩家 A 需要把一批高价值货物从系统 X 运到系统 Y。

玩家 A 创建一个 `Escort` 请求：

1. 填写起点、终点、预计出发时间。
2. 填写货物风险等级和护航赏金。
3. 选择需要的人数或编队规模。
4. 锁定报酬。

接单者到齐后：

1. 工单进入 `In Progress`。
2. 护航完成后，请求人确认。
3. 系统放款。

### 8.3 场景 C：Recovery

玩家或 Tribe 在战斗后需要回收资产和重组 fleet。

Recovery 请求可以包含：

1. 残留资产位置
2. 需要回收的物品类型
3. 报酬
4. 时间窗口
5. 是否需要额外护航

MVP 阶段只保留最小版本，不要求复杂的自动验证。

## 9. MVP 定义

### 9.1 MVP 只做什么

MVP 只做 `SOS Rescue` 主线，并允许数据模型兼容 `Escort` 和 `Recovery`。

必须完成的能力：

1. 发布 SOS 请求
2. 锁定奖励
3. 接单
4. 更新工单状态
5. 请求人确认完成
6. 自动结算奖励
7. 展示历史记录

### 9.2 MVP 不做什么

1. 不做自动检测玩家是否真的“缺油”或“过热”
2. 不做全自动链上事实校验
3. 不做复杂仲裁
4. 不做多方分账
5. 不做站内聊天

## 10. 核心体验原则

1. `一眼懂`
评委和玩家在首页就能看懂“有人求救，别人接单，完成后拿钱”。

2. `先人工确认，再追求自动化`
MVP 阶段宁可手动确认完成，也不要为了炫技做不稳的自动验证。

3. `状态必须清晰`
每个工单必须有清楚的生命周期，不能让用户猜当前到哪一步了。

4. `结算必须可信`
钱要么在 escrow 里，要么已经放给服务提供者，不能靠口头承诺。

## 11. 核心实体

### 11.1 Request

字段建议：

- `request_id`
- `job_type`: `rescue` / `escort` / `recovery`
- `requester_wallet`
- `start_system`
- `target_system`
- `hazard_level`
- `reward_amount`
- `deadline`
- `requirements`
- `status`

### 11.2 Assignment

- `assignment_id`
- `request_id`
- `responder_wallet`
- `accepted_at`
- `started_at`
- `completed_at`
- `completion_note`

### 11.3 Escrow

- `escrow_id`
- `request_id`
- `locked_amount`
- `token_type`
- `state`: `locked` / `released` / `refunded`

### 11.4 Reputation

MVP 只做最简化记录：

- `wallet`
- `jobs_completed`
- `jobs_cancelled`
- `jobs_failed`

MVP 不做复杂社交评分算法。

## 12. 工单生命周期

### 12.1 状态机

1. `Draft`
2. `Open`
3. `Accepted`
4. `In Progress`
5. `Awaiting Confirmation`
6. `Completed`
7. `Cancelled`
8. `Expired`
9. `Disputed`

### 12.2 MVP 允许的关键流转

- `Draft -> Open`
- `Open -> Accepted`
- `Accepted -> In Progress`
- `In Progress -> Awaiting Confirmation`
- `Awaiting Confirmation -> Completed`
- `Open -> Cancelled`
- `Open -> Expired`

`Disputed` 只预留状态，不要求比赛阶段完整实现仲裁流程。

## 13. 页面与信息架构

### 13.1 首页

首页只做两件事：

1. 告诉用户这是什么
2. 让用户立刻发布或接单

必须包含：

- 当前开放工单数量
- 三类服务介绍
- `Post SOS`
- `Browse Requests`

### 13.2 Request Board

工单列表页用于接单。

必须展示：

- 工单类型
- 起点/目标点
- 奖励
- 风险等级
- 剩余时间
- 当前状态

支持的筛选：

- `rescue`
- `escort`
- `recovery`
- 距离
- 奖励范围
- 风险等级

### 13.3 Request Detail

工单详情页必须明确：

- 谁发的
- 要解决什么问题
- 需要去哪
- 奖励多少
- 截止时间
- 当前有没有人接

如果当前用户是求助者，显示：

- 取消工单
- 确认完成

如果当前用户是服务者，显示：

- 接单
- 标记开始
- 标记已完成

### 13.4 Ops Console

这是一个轻量工作台，不是复杂后台。

它展示：

- 我发布的工单
- 我接的工单
- 我完成的工单
- 我的累计收入

## 14. 功能需求

### FR-01 钱包连接

- 系统必须支持连接 `EVE Vault`
- 未连接钱包时只能浏览公开工单，不能发单或接单

### FR-02 创建工单

- 用户必须能创建 `rescue` 工单
- MVP 必须支持填写位置、问题描述、赏金、截止时间
- 发布前必须锁定赏金

### FR-03 浏览和筛选工单

- 系统必须展示全部 `Open` 工单
- 用户必须可以按类型、风险、奖励筛选

### FR-04 接单

- 服务者必须能够接一个 `Open` 工单
- 工单被接后，其他人不能重复接单

### FR-05 工单状态推进

- 接单者必须可以把工单从 `Accepted` 更新为 `In Progress`
- 接单者完成后，必须可以进入 `Awaiting Confirmation`
- 请求人必须可以确认完成

### FR-06 奖励锁定与结算

- 请求发布时必须锁定赏金
- 工单完成确认后必须自动放款
- 取消或过期时必须按规则退款

### FR-07 历史记录

- 每个钱包必须能看到自己的发单和接单历史
- 至少保留最近 `20` 条记录

### FR-08 最小声誉

- 系统必须展示一个用户完成过多少单
- MVP 不做复杂评分，只做可见历史

### FR-09 失败与取消

- 工单过期时必须进入 `Expired`
- 未被接单的工单允许请求人取消
- 已接单工单的取消需要简单限制，避免恶意跳单

### FR-10 演示友好

- 每个关键动作都必须有明确的 loading、成功、失败反馈
- 每个链上操作都必须显示 digest 或可追踪结果

## 15. 非功能需求

### NFR-01 可理解性

- 第一次看 demo 的人必须在 `30` 秒内理解产品价值

### NFR-02 演示时长

- 一次完整演示应控制在 `90` 秒到 `3` 分钟之间

### NFR-03 可部署性

- 前端必须可独立部署为外部 dApp

### NFR-04 可扩展性

- 数据结构必须兼容从 `rescue` 扩展到 `escort`、`recovery`

### NFR-05 稳定性

- 任意一步链上失败时，页面不能进入不可恢复状态

## 16. 上链边界

这一节只写比赛阶段最值得上链、且最能讲清楚价值的部分。

### 16.1 上链部分

1. 工单创建时的奖励锁定
2. 工单接单状态
3. 工单完成后的放款
4. 退款状态
5. 最基础的公开历史记录

### 16.2 链下部分

1. 页面 UI
2. 地图与位置展示
3. 风险等级计算
4. 筛选和排序
5. 通知与提醒
6. 可选的聚合缓存

### 16.3 为什么这样切

因为 MVP 最重要的是保证：

- 钱不会凭空消失
- 工单状态是公开的
- 结算是可信的

而不是追求“地图和验证也全部上链”。

### 16.4 保守实现建议

比赛阶段建议用以下方式降低复杂度：

1. 用 escrow 保证赏金真实存在
2. 用请求人确认作为完成条件
3. 用过期退款做兜底
4. `Disputed` 先只保留状态，不做完整仲裁

## 18. Demo 脚本

### 18.1 Demo 目标

让评委看到这不是 another dashboard，而是一套能让玩家实际互相帮助并拿钱的系统。

### 18.2 推荐脚本

1. 主讲人先说背景：
   `这个玩家现在没油了，卡在外面，回不去。`
2. 玩家 A 连接钱包，发布一个 `SOS Rescue` 请求，锁定赏金。
3. 大屏展示工单出现在列表里。
4. 玩家 B 用另一个钱包接单。
5. 玩家 B 把工单推进到 `In Progress`。
6. 演示“已完成救援”，工单进入 `Awaiting Confirmation`。
7. 玩家 A 点击确认完成。
8. 系统自动放款，展示交易结果。
9. 最后展示双方历史记录和累计完成单数。

### 18.3 Demo 爽点

1. 玩家之间发生了真实服务关系
2. 赏金不是口头承诺，而是锁定资金
3. 完成后自动结算
4. 这件事不是抽象理论，而是针对 EVE Frontier 的真实痛点

## 19. 成功指标

比赛阶段的成功不看 DAU，而看这几个指标：

1. 是否能完整跑通一单 `SOS Rescue`
2. 是否能在 `Stillness` 或可验证环境里展示真实链上结果
3. 评委是否能在短时间内理解价值
4. 社区是否会说“这个我真会用”

## 20. 风险与应对

### 风险 1：自动验证太难

应对：

- MVP 直接采用请求人确认完成
- 不在比赛期硬做自动裁决

### 风险 2：工单真实性不足

应对：

- 用赏金锁定提高真实性
- 通过钱包历史和完成记录提供最基础声誉

### 风险 3：产品太像表单

应对：

- 首页直接展示 live requests
- 强化状态变化和结算结果
- demo 里一定使用两个真实钱包

### 风险 4：一次做三类工单太散

应对：

- MVP 只演 `SOS Rescue`
- `Escort` 和 `Recovery` 只作为同一框架下的扩展能力展示

## 23. 当前假设

本 PRD 基于以下假设：

1. 我们优先做一个外部 dApp，而不是游戏内 UI
2. 比赛阶段追求“真实可演示”优先于“完全自动验证”
3. 产品核心价值在于服务结算，而不是地图本身
4. `Escort` 和 `Recovery` 先作为二期能力，不抢占 MVP 资源

## 24. 一句话判断标准

如果用户看完 demo 之后会说：

`原来在 Frontier 里，救人这件事真的可以变成一个职业和市场`

那这个项目就站住了。
